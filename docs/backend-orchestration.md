# Backend Orchestration

The backend orchestration logic handles the full execution flow of sourcing runs, managing the interaction between the user-supplied ICP, the TinyFish automation layer, lead scoring, and session persistence.

The core of the orchestration logic is located in `apps/api/src/orchestrators/discoveryRun.ts`.

## Entry Point

A new discovery run is launched by calling `startDiscoveryRun()` from the `POST /api/runs` route handler. This function initializes the in-memory run state and immediately returns a `runId` to the caller (HTTP 202 Accepted). All subsequent orchestration happens asynchronously without blocking the HTTP response.

## Configuration

Runtime behaviour is controlled by environment variables, resolved by `resolveOrchestrationConfig()`:

| Variable | Default | Description |
|---|---|---|
| `TINYFISH_POLL_INTERVAL_MS` | `3000` | Milliseconds between batch status polls |
| `TINYFISH_ASYNC_RUN_TIMEOUT_MS` | `300000` (5 min) | Max wait time per async TinyFish run |
| `TINYFISH_INSPECTION_CONCURRENCY` | `4` | Max parallel website inspection runs |
| `TINYFISH_MAX_COMPANIES_TO_INSPECT` | `5` | Max candidates forwarded to inspection stage |

## Orchestration Stages

### 1. Run Mode Resolution
Before any TinyFish call is made, `resolveLiveMode()` checks for `TINYFISH_FORCE_MOCK` and `TINYFISH_API_KEY`. If mock mode is active, all TinyFish calls are replaced with deterministic sample data from `mocks/sampleLeads.ts`. The resolved mode (`live` or `mock`) and its reason are recorded on the run.

### 2. Directory Discovery Stage
The backend constructs a `DirectoryDiscoveryTask` via `createDirectoryDiscoveryTask()` (in `discoverCompanies.ts`), which maps the ICP's `targetMarket` and `location` fields to a Clutch.co directory URL using static path maps (`LOCATION_PATH_MAP`, `MARKET_PATH_MAP` in `prompts.ts`). The structured extraction goal is generated by `buildDirectoryGoal()`.

The task is submitted to TinyFish using `run-async`. The run is registered in the session store and the orchestrator enters a polling loop.

### 3. Candidate Extraction
Once the directory discovery run reaches a terminal state, `parseDirectoryDiscoveryResult()` is called to parse the raw JSON payload from TinyFish into an array of `DirectoryCandidate` objects. Each candidate carries: `companyName`, `websiteUrl`, `directoryUrl`, `location`, `shortDescription`, `primaryService`, `employeeRange`, `rating`, `listingFacts`, and `evidenceSnippet`.

The candidate list is capped at `maxCompaniesToInspect`.

### 4. Inspection Stage
The backend launches parallel website inspection tasks for each candidate, respecting the `inspectionConcurrency` limit. Each task is built by `createWebsiteInspectionTask()` using `buildWebsiteGoal()` (in `prompts.ts`), which instructs the agent to navigate the homepage, contact page, about page, and team page to extract structured data.

Each inspection is tracked as an `ActiveInspectionJob` in the session store. If a run times out, `createFailedInspection()` is called to produce a gracefully degraded `WebsiteInspection` with `inspectionStatus: "failed"` and populated `qualityNotes`.

### 5. Polling Loop
Both the discovery and inspection stages use a shared polling pattern: the backend calls `getTinyFishRunsByIds()` with a list of active run IDs (using the TinyFish `batch` endpoint) on each tick. Results are classified into `completed`, `failed`, and `pending` buckets (`PollResult`). Each completed run triggers its corresponding parse step. The loop continues until all runs are terminal or `asyncRunTimeoutMs` is exceeded.

### 6. Lead Scoring & Ranking
After all inspections are terminal, `processLeadCandidates()` (in `domain/leads/processing.ts`) feeds each `InspectedCandidate` through the TypeScript scorer (`scoreLeadInput()` in `ranking.ts`). The scorer computes four weighted sub-scores:

- **Fit score** — ICP market alignment, keyword matches, location match, company size match (base: 24)
- **Contactability score** — verified emails, discoverable team members (base: 12)
- **Quality score** — structured services, company summary, evidence count, page findings (base: 28)
- **Decision-maker score** — explicit role matches against ICP `decisionMakerRole` field (base: 10)

Scores are clamped to `[0, 100]`. `qualificationState` is derived from the total score threshold, and `priority` is set to `high` (≥72, qualified), `medium` (≥50, not unqualified), or `low`. Leads are ranked by total score descending.

### 7. Run Status Resolution
After scoring, the run's final status is resolved by `combineStatus()`. If any inspection failed or was skipped, the run is marked `partial` instead of `completed`. Both are terminal states that allow lead export.

### 8. Session Persistence
The completed run (leads, summary, telemetry) is written to SQLite via `persistRunSnapshotSafely()`, which wraps `persistDiscoveryRun()` in a try/catch so a persistence failure never crashes the orchestrator. On success, the session becomes accessible via `GET /api/sessions/:id`.

## Key Type Interfaces

```typescript
interface OrchestrationConfig {
  pollIntervalMs: number;
  asyncRunTimeoutMs: number;
  inspectionConcurrency: number;
  maxCompaniesToInspect: number;
}

interface InspectedCandidate {
  candidate: DirectoryCandidate;
  inspection: WebsiteInspection;
  inspectionRunIds?: string[];
}

interface ActiveInspectionJob {
  jobId: string;
  candidate: DirectoryCandidate;
  submittedAt: number;
  submittedAtIso: string;
}

interface PollResult {
  completed: TinyFishRunSnapshot[];
  failed: Array<{ snapshot?: TinyFishRunSnapshot; runId: string; error: string }>;
  pending: TinyFishRunSnapshot[];
}
```

## In-Memory State (runStore.ts)

Active runs are managed exclusively in memory during orchestration. `runStore.ts` exposes:

- `createRun()` / `getRun()` — lifecycle management
- `setStepStatus()` — updates individual step states (discovery, inspection, aggregation)
- `updateRunLeads()` / `updateRunState()` / `updateSummary()` — progressive state updates during the run
- `failRun()` / `finishRun()` — terminal state transitions
- `updatePushState()` — tracks Revon push status after run completion

## Telemetry

Every TinyFish run interaction is recorded via `telemetryStore.ts` and linked to the session. This includes per-run timings, credit usage, and status transitions. Telemetry is joined into the persisted session at finalization and exposed via `GET /api/telemetry`.
